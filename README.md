# Онкологический рабочий стол для Obsidian

Система предназначена для ведения рабочего процесса врача-радиотерапевта/онколога внутри Obsidian. Она объединяет карточки пациентов, оперативный рабочий стол, расписание лечения, напоминания, выписки, парсинг медицинских документов и выгрузку структурированных данных в единую DataviewJS-среду.

Проект не является самостоятельным веб-приложением. Это набор Markdown-точек входа, DataviewJS-runtime модулей и тестируемых CommonJS-библиотек для Obsidian vault.

## Для чего нужна система

Система закрывает ежедневные задачи отделения лучевой терапии:

- видеть пациентов в оперативке по статусам и датам;
- вести консультации, госпитализации, начало и окончание лечения;
- планировать разметку, оконтуривание, лечение, повторные этапы и выписку;
- вести карточку пациента с клиническими, административными и лечебными данными;
- считать фракции, дозу и текущий прогресс лучевой терапии;
- учитывать пропуски, переносы, выходные/праздники и режимы фракционирования;
- фиксировать выписку с расчетными показателями на дату выписки;
- вести напоминания, ЭЛН, ВК по ЭЛН, химиотерапию и контроль анализов;
- парсить документы через LLM и подтверждать найденные данные перед записью;
- поддерживать локальную базу пациентов для последующей аналитики.

## Основные функции

### Рабочий стол

Точка входа: `Рабочий стол.md`.

Рабочий стол показывает несколько вкладок:

- **Оперативка**: консультации, госпитализации, окончания лечения, активные напоминания, ВК, выписки, оконтуривание и разметки.
- **Планирование**: пациенты на этапах разметки, оконтуривания и ожидания лечения.
- **Лечение**: пациенты на активном курсе лучевой терапии.
- **Выписка**: пациенты, которым требуется подготовка или контроль выписки.

Поддерживаются фильтры, поиск, выбор даты просмотра, счетчики вкладок и карточки с клиническими бейджами.

### Карточка пациента

Карточка пациента открывается через DataviewJS runtime `Архив/Scripts/views/patient/view.js`.

В карточке доступны:

- основные данные пациента;
- диагноз, TNM, стадия, МКБ-10 и биомаркеры;
- параметры лучевого лечения;
- объемы лечения, последовательный буст, SIB и разные объемы;
- расчет текущей дозы и количества фракций;
- учет пропусков, переносов и ручных дат;
- химиотерапия и напоминания по введениям;
- ЭЛН и ВК по ЭЛН;
- прикрепление файлов и изображений;
- связанные клинические случаи;
- рабочий процесс консультация → лечение → выписка.

### Планирование и лечение

Модули расписания умеют:

- строить календарь фракций;
- учитывать пятидневку, выходные и праздники;
- поддерживать режимы через день и два раза в день;
- учитывать ручные добавленные/исключенные даты;
- считать фактически выполненные фракции;
- разделять дозу по объемам и не смешивать независимые объемы лечения;
- учитывать последовательный буст отдельно от одновременного лечения.

### Выписка

При выписке фиксируется снимок состояния пациента на дату выписки:

- количество выполненных фракций;
- текущая суммарная доза;
- данные по объемам;
- введенная химиотерапия;
- клинические и административные поля;
- дата и статус выписки.

Эти значения используются при выгрузке в базу и не должны пересчитываться задним числом при последующих изменениях расписания.

### Напоминания, ЭЛН и ВК

Система поддерживает:

- обычные напоминания по пациентам;
- напоминания по фракциям;
- ВК по ЭЛН;
- расчет даты следующего ВК;
- отметку выполненных напоминаний;
- контроль активных и просроченных задач на рабочем столе.

Выполненные ВК по ЭЛН не выводятся на рабочий стол.

### AI-парсер документов

Парсер помогает извлекать структурированные данные из медицинских текстов:

- ФИО, дата рождения, СНИЛС, телефон;
- диагноз, МКБ-10, TNM, стадия;
- гистология, локализация, молекулярные маркеры;
- предшествующее лечение;
- химиотерапия, гормонотерапия, таргетная терапия, иммунотерапия;
- лабораторные значения;
- административные поля.

После парсинга данные проходят ревью:

- конфликтные поля показываются первыми;
- строки сгруппированы по смыслу;
- для каждой строки видно "было" и "станет";
- безопасные поля с уверенностью 100% автоподтверждаются и сворачиваются;
- врач может принять новое значение, оставить старое или объединить текст.

Парсер поддерживает OpenRouter и LiteLLM-прокси. Ключи API хранятся локально в Obsidian/localStorage, в репозиторий их добавлять нельзя.

### База данных

Слой базы отвечает за:

- формирование строки пациента;
- сохранение структурированных полей;
- синхронизацию выписанных пациентов;
- сохранение QLQ и административных данных;
- стабильный `ID_пациента`;
- защиту ручных значений от случайной перезаписи.

### Корневые файлы

- `Рабочий стол.md` — основная точка входа рабочего интерфейса. Загружает `Архив/Scripts/views/desktop`.
- `Шаблон пациента.md` — архивный stub. Новые карточки создаются через актуальный шаблон.
- `README.md` — описание проекта.
- `Пациенты/` — активные карточки пациентов на лечении или в рабочем маршруте.
- `Консультации/` — пациенты и документы на этапе консультации.
- `Не начали/` — пациенты, которые еще не начали лечение.
- `Выписка/` — пациенты и материалы на этапе подготовки выписки.

### `Архив/Scripts/views`

DataviewJS-точки входа:

- `views/desktop/view.js` — рабочий стол.
- `views/desktop/view.css` — стили рабочего стола.
- `views/patient/view.js` — карточка пациента.
- `views/patient/view.css` — стили карточки пациента.

### `Архив/Scripts/src/shared`

Тестируемые ядра и UI-модули:

- `desktop-core.cjs` — общая логика рабочего стола.
- `desktop-render.cjs` — рендер вкладок рабочего стола.
- `desktop-calendar-core.cjs` — календарь и выбор дат.
- `desktop-discharge-core.cjs` — логика выписки.
- `desktop-parser-ui.cjs` — AI-парсер и ревью данных.
- `patient-model-core.cjs` — модель пациента и расчет лечения.
- `schedule-core.cjs` — расписание фракций.
- `database-core.cjs` — структура и синхронизация базы.
- `diagnosis-core.cjs` — нормализация диагноза, TNM, стадий и биомаркеров.
- `patient-reminder-core.cjs` — напоминания, ЭЛН и ВК.
- `patient-workflow-core.cjs` — workflow пациента.
- `storage-frontmatter-core.cjs` и `write-layer-core.cjs` — безопасная запись frontmatter.

### `Архив/Scripts/modules`

Runtime-обертки для загрузки shared-кода из DataviewJS. Они связывают Obsidian API, `dv`, `app`, frontmatter и общие модули.

### `Архив/Scripts/templates`

- `patient-note.md` — актуальный шаблон новой карточки пациента.

### `Архив/Scripts/tests`

Набор `.cjs` тестов для ядра, runtime-интеграций и регрессий. Тесты запускаются обычным Node.js.

### `Архив/Scripts/tools`

Служебные скрипты для проверки, восстановления и миграции DataviewJS-кода.

## Требования

- Obsidian.
- Плагин Dataview с поддержкой DataviewJS.
- Node.js 20+ для запуска тестов.
- Доступ к OpenRouter или LiteLLM нужен только для AI-парсера.

## Установка в Obsidian

1. Скопируйте содержимое репозитория в Obsidian vault или в отдельную папку внутри vault.
2. Убедитесь, что плагин Dataview включен и DataviewJS разрешен.
3. Откройте `Рабочий стол.md`.
4. Для создания пациента используйте кнопку/действие создания пациента на рабочем столе. Новый файл будет создан по шаблону `Архив/Scripts/templates/patient-note.md`.

## Настройка AI-парсера

Парсер работает через OpenRouter или LiteLLM-прокси.

Ключи не хранятся в файлах проекта. Они записываются локально в настройках интерфейса/`localStorage`.

Важно:

- не коммитьте API-ключи;
- не добавляйте в репозиторий реальные медицинские документы;
- перед публикацией проверьте, что в vault нет персональных данных пациентов.

## Запуск тестов

Установите зависимости:

```bash
npm install
```

Запустите все тесты:

```bash
npm test
```

Тестовый скрипт последовательно запускает все файлы `Архив/Scripts/tests/*.cjs`.

Для запуска отдельного теста:

```bash
node "Архив/Scripts/tests/shared-desktop-core.cjs"
```

## Подготовка к публикации на GitHub

В репозиторий можно добавлять:

- `Рабочий стол.md`;
- `Шаблон пациента.md`;
- `README.md`;
- `package.json`;
- `.gitignore`;
- `.github/workflows/tests.yml`;
- папку `Архив/Scripts`;
- справочники без персональных данных, например `Архив/Ускорители.md`;
- пример справочника врачей `Архив/Врачи.example.md`.

Не добавляйте:

- реальные карточки пациентов;
- выписки и консультации с персональными данными;
- вложения, сканы, фото и медицинские документы;
- API-ключи;
- `node_modules`;
- локальные кэши Obsidian/Codex.
- локальный `Архив/Врачи.md`, если в нем указаны реальные сотрудники.

## Безопасность данных

Проект может использоваться рядом с медицинскими персональными данными. Перед публикацией на GitHub обязательно проверьте:

- нет ли ФИО, СНИЛС, телефонов, дат рождения и номеров документов;
- нет ли вложений с медицинскими изображениями или сканами;
- нет ли реальных выписок, консультаций и лабораторных документов;
- нет ли API-ключей в Markdown, JavaScript, JSON или истории git.

Рекомендуемый порядок перед первым публичным push:

```bash
git status
git diff --cached
git grep -n "api_key\|sk-\|СНИЛС\|Дата_рождения\|Номер_телефона"
```

## Архитектурные принципы

- Точки входа в Obsidian остаются тонкими.
- Основная логика выносится в `src/shared` и покрывается Node.js тестами.
- Runtime-обертки в `modules` адаптируют shared-код к Obsidian API.
- Запись frontmatter должна идти через общий write/storage слой.
- Логика расписания, дозы, выписки, базы и парсера должна быть проверяемой вне Obsidian.
- Врач подтверждает значимые изменения данных после AI-парсинга.

## Ограничения

- Система зависит от Obsidian и DataviewJS.
- UI не предназначен для запуска как самостоятельный сайт.
- AI-парсер не является источником истины: итоговые данные должны подтверждаться пользователем.
- Медицинские расчеты и формулировки требуют клинической проверки перед использованием в документах.

## Статус

Проект находится в активной разработке. Основная архитектура уже разделена на рабочий стол, карточку пациента, shared-ядра, runtime-модули и тесты.
